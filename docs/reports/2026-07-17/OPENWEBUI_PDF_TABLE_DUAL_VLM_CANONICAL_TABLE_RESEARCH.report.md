# PDF table: 8% padding и dual-VLM canonical JSON

Дата: 2026-07-17

Обновлено после дополнительных прогонов sparse/bounds prompt, двух широких таблиц и literal-complete prompt v4.

## Короткий вывод

8% добавлены как единое правило для всех таблиц: от исходной рамки детектора отступаем на 8% ширины страницы слева и справа и на 8% высоты страницы сверху и снизу, затем ограничиваем результат границами страницы. Индивидуальной настройки для отдельных таблиц нет.

На пяти управляемых таблицах с точной истиной Gemini и OpenAI дали один и тот же правильный канонический JSON во всех 5 случаях. Ложного совпадения не было. На девяти реальных PDF полного совпадения не было ни разу: 0/9 по структуре, 0/9 по содержимому и 0/9 полностью. Это означает, что правило автоматического принятия по полному consensus на реальных таблицах пока не получило покрытия.

Дополнительные тесты уточнили причину, но не изменили итог. Промпт про разреженные столбцы помог моделям замечать отдельные валютные колонки, а literal-complete prompt помог OpenAI сохранить служебную строку `1...16` широкой таблицы. Однако на таблице с долларом OpenAI всё равно удалил два видимых `$`, а на широкой таблице вернул все 12 строк, но объявил `row_count=11`. Значит, более строгий промпт улучшает отдельные случаи, но не обеспечивает полноту и валидный детерминированный JSON.

Повторный запуск на тех же PNG также изменил часть ответов каждого провайдера. Поэтому temperature 0 здесь не равен гарантии байт-в-байт или канонической повторяемости.

Практический вывод изменился не в сторону автоматического склеивания, а в сторону явного протокола неопределённости. Gemini целесообразно временно использовать как `primary_provider`, потому что в текущих прогонах он стабильнее соблюдал контракт. Но это не «истинный оракул»: accuracy на реальных PDF без авторитетной разметки неизвестна. OpenAI остаётся независимым verifier.

Если оба валидных канонических ответа совпали, фиксируем `consensus`. Если они различаются, сохраняем таблицу Gemini как primary-кандидат, полный ответ OpenAI и детерминированный машинный diff; ставим `review_required=true`. Если OpenAI нарушил контракт, это отдельный статус `verifier_invalid`, а не обычное разногласие. Если невалиден Gemini, результат нельзя автоматически принимать.

Итоговый вердикт: `DUAL_VLM_CANONICAL_TABLE_CONSENSUS_PROMISING_BUT_NOT_PROVEN`.

Это исследовательский результат, не готовность production-пайплайна.

## Что именно стало выходом нормализатора

Основной объект — логическая сетка таблицы, а не набор финансовых фактов и не пары ключ–значение. В контракте нет координат, ролей header/body/total, типов number/date/currency и финансовой интерпретации.

```json
{
  "schema_version": "broker_reports_canonical_table_v1",
  "table_id": "example",
  "row_count": 2,
  "column_count": 2,
  "cells": [
    {
      "row_index": 0,
      "column_index": 0,
      "row_span": 1,
      "column_span": 2,
      "content_state": "present",
      "source_text": "Summary"
    },
    {
      "row_index": 1,
      "column_index": 0,
      "row_span": 1,
      "column_span": 1,
      "content_state": "empty",
      "source_text": ""
    },
    {
      "row_index": 1,
      "column_index": 1,
      "row_span": 1,
      "column_span": 1,
      "content_state": "present",
      "source_text": "1,000"
    }
  ]
}
```

Объединённая ячейка хранится один раз в своей верхней левой позиции со span. Каждое место сетки должно быть покрыто ровно одной ячейкой. Состояния закрыты: `present`, `empty`, `unreadable`. Строка `source_text` не превращается в число, дату, валюту или бизнес-понятие.

Для сравнения применяется только замороженная техническая нормализация пробелов и Unicode NFKC. Исходные ответы провайдеров сохранены отдельно, поэтому отличие форматирования не скрывается.

## Как обеспечена независимость

Оба провайдера получили байт-в-байт один PNG, один логический prompt и одну каноническую схему. На запрос — один preflight и один generate; скрытых повторов, failover, третьей VLM и передачи ответа соседа не было. Эталон открывался scorer-ом только после seal терминала.

Gemini использовал техническую проекцию JSON Schema. Проверка показала, что обязательные поля, enum и nullability остались эквивалентны. Сравниваемая единица — `model + provider API + schema adapter`; изолированная причинность модели не доказана.

## 8% padding и детектор

Все 14 кропов воспроизвелись байт-в-байт: да.

На реальном корпусе 8%-кроп содержит все 9 рамок из старого чернового референса. Это полезная геометрическая диагностика, но не accuracy: старый референс явно помечен `human_reviewed: false`. Поэтому надёжность детекции и полнота реального crop в строгом смысле пока не доказаны человеком.

## Точная истина: управляемые таблицы

Корпус: 5 заранее сгенерированных таблиц, 45 логических ячеек. Он включает обычную сетку, merged header, пустые ячейки, borderless-таблицу, исходные форматы `(50)`, `$20.00`, `01/02/25` и закрытую нечитаемую ячейку.

| Метрика | Gemini | OpenAI |
|---|---:|---:|
| Валидный контракт | 5/5 | 5/5 |
| Точная структура | 5/5 | 5/5 |
| Полностью правильная таблица | 5/5 | 5/5 |
| Правильное содержимое ячеек | 45/45 | 45/45 |
| Пропущенные / лишние ячейки | 0 / 0 | 0 / 0 |

| Dual-VLM метрика | Результат |
|---|---:|
| Структурный consensus | 5/5 |
| Content consensus | 5/5 |
| Full-table consensus | 5/5 |
| P(correct \| full consensus) | 100.0% |
| Ложный consensus | 0 |
| Покрытие automatic acceptance | 100.0% |

На этой малой точной выборке правило `accept only full consensus` не приняло ни одной неправильной таблицы. Но пяти простых контролей недостаточно для production-вывода.

## Реальные PDF: только диагностика consensus

Accuracy здесь не считается: нет авторитетного построчного референса. Можно честно сказать только, совпали ли два ответа.

| Метрика | Результат |
|---|---:|
| Gemini: валидный контракт | 9/9 |
| OpenAI: валидный контракт | 8/9 |
| Structural consensus | 0/9 |
| Content consensus | 0/9 |
| Full-table consensus | 0/9 |

Минимальные причины расхождения:

- разное число строк: 5;
- merged cell против разделённых ячеек: 3;
- текст назначен другой ячейке: 0;
- contract failure: 1.

В зафиксированном основном benchmark run contract failure у OpenAI: 1 (`drivewealth_p07:table_1`). Модель указала ячейку с `row_index` за пределами собственного `row_count`. Скорер не исправлял это и не выдавал за ошибку чтения.

## Дополнительные тесты: как развивалась гипотеза промпта

Первый промпт просил построить устойчивую логическую сетку. После разбора расхождений гипотеза уточнялась небольшими шагами:

1. Разреженное содержимое в одной вертикальной полосе считается признаком отдельного столбца, даже если большинство его ячеек пусты.
2. Короткие маркеры, например `$`, должны оставаться отдельной ячейкой, когда занимают собственную выровненную полосу.
3. `row_count` и `column_count` должны включать полную сетку, а все anchors и spans обязаны лежать внутри объявленных границ.
4. В literal-complete v4 добавлено прямое требование переносить каждый видимый токен без сокращения: строки нумерации, секционные заголовки, сноски, надстрочные знаки, повторяющиеся подписи и отдельные символы.

Это важная последовательность: сначала проверялась геометрическая гипотеза о разреженном столбце, затем валидность размеров, затем полнота текста. В каждом шаге менялся только prompt-контракт, а crop и JSON Schema оставались общими для двух провайдеров.

### Повторный прогон девяти реальных таблиц

| Вариант промпта | Gemini: валидный контракт | OpenAI: валидный контракт | Full consensus | Contract failures |
|---|---:|---:|---:|---:|
| Sparse-column без усиленной проверки границ | 8/9 | 7/9 | 0/9 | 3 |
| Sparse-column + обязательные bounds | 9/9 | 8/9 | 0/9 | 1 |

Bounds-условие уменьшило число формально невалидных ответов, но не сблизило модели: full consensus остался 0/9. Число случаев с разным `row_count` выросло с 3 до 6. Это не обязательно ухудшение чтения: часть прежних contract failures превратилась в валидные, но разные представления. Однако для автоматического принятия результата разницы нет — consensus по-прежнему отсутствует.

### Две широкие таблицы, prompt v3

| Таблица | Gemini | OpenAI | Что показал тест |
|---|---|---|---|
| 18 столбцов | валидно, 8×18, 114 anchors | объявлено 7×18, 110 anchors, присутствует `row_index=7`; contract failure | обе модели увидели 18 столбцов, но OpenAI ошибся на единицу в метаданных строки |
| 16 столбцов, сделки | валидно, 12×16, 177 anchors | валидно, 11×16, 161 anchors | OpenAI исключил видимую строку нумерации `1...16`; после сдвига остальные строки совпали, кроме потерянного маркера сноски `5` в заголовке статуса |

Вторая таблица особенно полезна аналитически. Основные строки сделок и их числовые значения две VLM прочитали практически одинаково. Расхождение возникло не в финансовых числах, а в политике включения служебной строки и сноски. Следовательно, задача состоит не только в OCR, но и в одинаковом определении того, что считается содержимым таблицы.

### Literal-complete prompt v4: таблица с долларом

На изображении видны четыре строки, три выровненные полосы и два отдельных знака `$`.

| Провайдер | Результат |
|---|---|
| Gemini | валидно, 4×3, 12 anchors; отдельная средняя колонка, `$` в первой и последней строках, явные пустые ячейки между ними |
| OpenAI | валидно, 4×3, 8 anchors; левая ячейка каждой строки растянута на два столбца, оба `$` отсутствуют |

Оба ответа прошли JSON-контракт, но consensus отсутствует: 4 различия `merged cell versus separate cells` и 4 отсутствующие у OpenAI средние ячейки. Это показывает ограничение схемы: структурная валидность не умеет доказать, что модель перенесла все видимые токены. Даже максимально прямое текстовое требование не гарантировало сохранение `$`.

### Literal-complete prompt v4: широкая таблица на 16 столбцов

Literal-условие улучшило полноту OpenAI: теперь он сохранил строку `1...16`, секционный заголовок и все девять строк сделок. Оба raw-ответа содержат по 177 anchors.

| Провайдер | Результат |
|---|---|
| Gemini | валидно, `row_count=12`, `column_count=16`, максимальный `row_index=11` |
| OpenAI | `row_count=11`, `column_count=16`, но максимальный `row_index=11`; 16 ячеек последней строки выходят за объявленную границу |

OpenAI поэтому остановлен как `terminal_contract_failure`. По raw-содержимому он почти повторил Gemini; сохраняется потеря видимого маркера сноски `5` в `Статус сделки⁵`. Но выдавать такой ответ за consensus нельзя: валидатор обязан оценивать реально объявленный контракт, а не мысленно исправлять `row_count`.

### Что в итоге дала работа с промптом

- Гипотеза о разреженных колонках здравая: она описывает реальную геометрию таблиц и должна остаться в prompt-контракте.
- Literal-complete формулировка полезна: на широкой таблице она вернула ранее потерянную строку нумерации.
- Одного промпта недостаточно: `$` всё равно был удалён, сноска потеряна, а `row_count` остался ошибочным.
- Ошибки делятся минимум на три класса: чтение токена, выбор логической структуры и согласованность собственных метаданных JSON. Один общий показатель «VLM ошиблась» скрывает эту разницу.
- Схема и prompt проверяют разные вещи. Schema хорошо останавливает выход индекса за границы, но не может увидеть пропущенный символ на изображении.
- Нельзя на основании этих тестов объявить Gemini истинно точнее на реальных PDF. Можно сказать только, что в текущей выборке он чаще давал валидный и визуально полный primary-кандидат.

|---|---:|
| Gemini / controlled | 5/5 |
| OpenAI / controlled | 5/5 |
| Gemini / real PDF | 7/9 |
| OpenAI / real PDF | 1/7 |

У Gemini изменились реальные таблицы: `betterment_p04:table_0`, `ibkr_annual_p11:table_1`. У OpenAI изменились: `betterment_p04:table_0`, `ibkr_annual_p11:table_1`, `ibkr_midyear_p03:table_1`, `moomoo_annual_p14:table_1`, `moomoo_annual_p14:table_2`, `moomoo_midyear_p10:table_0`, `moomoo_midyear_p10:table_1`. ## Конкретные примеры

- Правильный полный consensus: `controlled_simple_ruled` — без расхождения.
- Структурное расхождение: `betterment_p04:table_0` — {"cell": null, "class": "different_row_count", "left": 21, "right": 15}.
- Пропуск ячейки одним провайдером: `betterment_p04:table_0` — {"cell": [11, 2], "class": "one_provider_omitted_cell", "left": null, "right": {"column_index": 2, "column_span": 1, "content_state": "empty", "row_index": 11, "row_span": 1, "source_text": ""}}.
- Provider contract failure: `drivewealth_p07:table_1` — {"cell": null, "class": "provider_contract_failure", "left": "completed", "right": "terminal_contract_failure"}.
- Чистое content-only расхождение: не наблюдалось.
- Ложный consensus: не наблюдалось.

Чистого content-only расхождения, ложного consensus и одинаковой ошибки двух провайдеров на точном корпусе не наблюдалось. Это не доказывает, что они невозможны.

## Ответы на главные вопросы

- Один VLM нашёл девять ожидаемых областей, а единый 8% padding геометрически накрыл все девять черновых рамок. Строго назвать детекцию и crop надёжными пока нельзя: реальный reference не прошёл human review.
- Gemini правильно восстановил 5/5 управляемых таблиц. Его accuracy на реальных PDF неизвестна.
- OpenAI правильно восстановил 5/5 управляемых таблиц. На реальных PDF он дал валидный контракт в 8/9 случаях; accuracy без истины неизвестна.
- На реальных таблицах модели согласились по структуре 0/9, по содержимому 0/9, полностью 0/9.
- Полный consensus был правильным 5/5 раз только на управляемом корпусе. Ложных совпадений там не было.
- Нельзя сказать, какая модель точнее в реальных разногласиях: для этого нужна авторитетная разметка.
- Дополнительные тесты подтвердили, что расхождения бывают материальными даже при валидной схеме: OpenAI удалил два `$` в маленькой таблице. Бывают и почти техническими: на широкой таблице он вернул все видимые строки, но занизил `row_count` на единицу.
- Literal-complete prompt улучшил полноту широкой таблицы, но не создал гарантию буквального переноса. Поэтому дальнейшее бесконечное усиление текста промпта без изменения контракта и контроля качества имеет убывающую отдачу.
- Второй VLM даёт материальную страховку: он делает расхождение видимым и не позволяет молча назвать primary-ответ проверенным. Но сам по себе verifier не устанавливает истину.
- Full consensus достаточен как условие принятия внутри малого управляемого корпуса, но не доказан для реальных брокерских PDF.
- Temperature 0 не обеспечил повторяемость реальных JSON: оба провайдера меняли часть своих канонических таблиц между двумя одинаковыми запусками.
- Gemini можно назначить текущим primary по операционной стабильности контракта, но слово «оракул» для него слишком сильное. Этот выбор должен быть конфигурацией и версионироваться вместе с моделью, prompt и schema.

## Что передавать следующему агенту

Предлагается хранить не только таблицу, но и решение dual-VLM уровня. Базовая таблица остаётся `broker_reports_canonical_table_v1`; поверх неё нужен небольшой envelope:

```json
{
  "schema_version": "broker_reports_dual_vlm_table_decision_v1",
  "status": "disagreement",
  "primary_provider": "gemini",
  "table": {
    "schema_version": "broker_reports_canonical_table_v1"
  },
  "verification": {
    "provider": "openai",
    "contract_valid": true,
    "table": {
      "schema_version": "broker_reports_canonical_table_v1"
    }
  },
  "diff": {
    "structural": [],
    "cells": []
  },
  "review_required": true,
  "evidence": {
    "crop_sha256": "...",
    "prompt_contract_version": "dual_vlm_canonical_table_normalizer_v4",
    "schema_sha256": "..."
  }
}
```

Статусы должны иметь однозначную машинную семантику:

- `consensus`: оба ответа валидны и равны после замороженной канонизации; в `table` лежит общий результат, `review_required=false`;
- `disagreement`: оба ответа валидны, но различаются; Gemini хранится как primary-кандидат, OpenAI и точный diff не теряются, `review_required=true`;
- `verifier_invalid`: Gemini валиден, OpenAI нарушил контракт или не завершил запрос; это технически не consensus и не обычное содержательное расхождение;
- `primary_invalid`: Gemini невалиден; никакую таблицу нельзя автоматически объявлять основной, даже если OpenAI вернул корректный JSON.

Следующий агент может получать envelope и видеть семантическое содержимое Gemini даже при разногласии, но обязан учитывать `status` и `review_required`. Автоматические или необратимые действия допустимы только по отдельно утверждённой политике; сам факт наличия `table` не означает, что она проверена.

Diff должен вычисляться обычным детерминированным кодом по двум каноническим сеткам. Нельзя поручать одной VLM объяснять или исправлять ответ другой: это вернёт недетерминированность и скроет первичное свидетельство.

## Минимальная оправданная архитектура

1. Один замороженный VLM-детектор области таблицы.
2. Один глобальный crop policy: 8% каждой стороны страницы, без per-table tuning.
3. Один нейтральный canonical-table контракт и один логический prompt для Gemini и OpenAI.
4. Один frozen prompt snapshot на таблицу: id/command, версия и hash фиксируются до вызова первой VLM и одинаково применяются к обеим.
5. Gemini — конфигурируемый `primary_provider`, OpenAI — независимый verifier. Это routing policy, а не утверждение об истинности модели.
6. Сравнение, статусы и diff принадлежат детерминированному orchestration-коду, а не провайдерам.
7. При consensus хранится общий канонический результат. При disagreement хранятся primary, verifier и diff; никакого silent merge, исправления одной моделью другой или majority vote.
8. Raw-ответы, canonical outputs, crop hash, model ids, prompt hash и schema hash сохраняются как evidence вне компактного бизнес-объекта.

## Где должен жить промпт

Сейчас prompt v4 намеренно захардкожен в research harness. Для эксперимента это правильно: старый запуск можно воспроизвести по точной версии текста и hash, а изменение prompt не зависит от состояния внешней базы OpenWebUI.

Для production рекомендуется гибридный путь:

1. Канонический Markdown prompt хранится в репозитории и проходит review как код.
2. Deploy/reconcile публикует его в штатное хранилище Prompt OpenWebUI.
3. Runtime resolver один раз получает разрешённый prompt по id/command, проверяет ACL и фиксирует id, version/hash.
4. Один и тот же snapshot передаётся обоим provider adapters.
5. JSON Schema остаётся кодовым контрактом, а не редактируемым текстом Prompt.

Не следует поддерживать разные по смыслу Gemini- и OpenAI-промпты или молча откатываться на hardcoded-версию при ошибке managed prompt. Иначе сравниваться будут уже разные задания, а доказательство consensus потеряет смысл.

## Предлагаемые следующие шаги

1. Реализовать версионированный decision envelope и четыре статуса без изменения самих provider calls.
2. Назначить Gemini primary только конфигурацией; сохранить возможность смены primary после human-reviewed benchmark.
3. Рассмотреть `canonical_table_v2`, где `row_count` и `column_count` вычисляются кодом из anchors и spans, а модель их не объявляет. Это уберёт наблюдавшийся off-by-one, но должно быть отдельной миграцией контракта, а не скрытым ремонтом уже полученного JSON.
4. Добавить проверку полноты видимых токенов, если будет независимый token inventory/OCR evidence. Без такого источника schema не способна обнаружить пропавший `$` или сноску.
5. Разметить human-reviewed truth минимум для таблицы с долларом, обеих широких таблиц и девяти исходных реальных кропов.
6. После разметки отдельно измерить: точность primary, recall verifier-а на ошибках primary, долю consensus, ложный consensus и долю disagreement, где primary всё же был правильным.
7. Только после этого решать, можно ли передавать `disagreement` следующему агенту без ручного review в низкорисковых сценариях.
