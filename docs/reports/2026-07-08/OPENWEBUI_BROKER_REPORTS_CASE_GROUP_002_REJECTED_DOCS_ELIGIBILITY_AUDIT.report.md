## Root Cause

The 2 accepted documents were the 2 CSV operation tables. They had
`machine_readable_table=true`, taxonomy `operations_table`, no blockers, and
entered the included Gate 2 refs.

The other 14 documents did not fail because Gate 2 handoff excluded broker
reports. `source_broker_report`, `dividends_report`, `fees_report` and
`withholding_report` are already allowed source document classes. The failures
were upstream:

1. PDF profiler false-positive OCR routing.
   The live stdlib PDF heuristic did not decode compressed PDF content streams
   and marked text-layer PDFs as `text_layer=no`. Because image markers were
   also present, it emitted `raster_requires_ocr_or_review`.

2. HTML table evidence loss.
   The live HTML profiler detected `table_candidate=true`, but persisted only a
   text excerpt slice. It did not persist private table slices, and
   `machine_readable_table=false` remained in the profile.

3. Taxonomy did not promote PDF/HTML broker reports.
   Live taxonomy saw only weak bounded snippets and produced
   `unknown_or_needs_review` for all 8 PDFs and 4 HTML files. Safe registry
   roles say these are source candidates, but the live normalizer did not use
   registry role hints during file-only OpenWebUI intake.

4. Eligibility and handoff behaved as written.
   They routed based on taxonomy and blockers. No evidence showed a year,
   account, broker or handoff-filter mismatch.

## PDF Audit Answer

The live classifier incorrectly treated text-layer PDFs as OCR candidates.

Secondary parser aggregate probe over the 8 PDFs:

| Metric | Value |
|---|---:|
| PDF count | 8 |
| text layer present | 8 |
| zero-text PDFs | 0 |
| PDFs with images | 8 |
| total extracted text chars | 761,341 |

This means the live heuristic confused mixed text+image PDFs with raster/scans.
The real distinction should be:

- `text_layer_pdf`
- `mixed_pdf_with_text`
- `raster_pdf_or_image_only`
- `pdf_requires_parser_review`

Only the third category should enter OCR-required routing while OCR is disabled.

## Fix Implemented

Implemented in Gate 1 backend and rebuilt the bundled OpenWebUI Pipe:

- PDF profiler now decodes Flate streams and hex text strings using stdlib
  only.
- PDF profile now emits text-layer evidence counts and `pdf_content_kind`.
- PDF profiler no longer emits OCR blocker when a text layer/text operators are
  detected, even if image markers exist.
- HTML profiler now persists bounded private `table_rows` slices for detected
  tables.
- HTML profile now records table counts and sets
  `machine_readable_table=true` when table rows exist.
- Taxonomy signals were broadened for common broker/source-table terms.
- Contract updated to document PDF/HTML eligibility behavior.

No new runtime dependency, OCR, VLM, pandas, tax logic, declaration logic or
XLS export was added.

So the fix is meaningful but conservative. It removes the wrong OCR routing and
preserves HTML/PDF private evidence. It does not automatically expand the Gate 2
accepted subset beyond the 2 CSVs, because the live file-only taxonomy still
lacks a trusted source-role decision for most PDF/HTML broker reports.

To expand the reduced subset safely, we need one of these explicit policy
decisions:

1. allow customer-approved safe registry role hints to participate in Gate 1
   source eligibility for private case packages; or
2. add a stronger non-OCR PDF/HTML parser dependency to runtime and prove it in
   the bundled Pipe; or
3. keep PDF/HTML broker reports in `unknown_role_requires_review` until a
   specialist explicitly approves them as Gate 2 source evidence.
