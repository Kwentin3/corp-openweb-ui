# Broker Reports — отчёт о проделанной работе

Дата: 23 июля 2026 года

Итоговый статус: `NOT_CLOSED`

## Что было сделано

Мы проверили весь путь обработки брокерского отчёта — от загрузки PDF в
OpenWebUI до подготовки данных для ответа пользователю.

В ходе работы:

- подтверждена корректность упрощённого semantic JSON для визуальных таблиц;
- сохранена схема Gemini master с возможностью контролируемого fallback;
- проверена изоляция Broker Reports от Knowledge, RAG и векторного поиска;
- выполнен полный набор тестов Broker Reports: 1 147 успешно, 20 пропущено,
  ошибок нет;
- проверены приватный intake, WorkloadAuthority, ArtifactStore и Gate 2;
- исправлено определение текущего DCP по авторизованному чату;
- исправлена потеря исходного чата при переключении на обе модели Gate 2;
- обе runtime-коррекции выпущены атомарно с проверкой rollback;
- системный Chrome подтвердил, что запрос, ответ и адрес страницы остаются в
  одном чате;
- подтверждено отсутствие изменений в Knowledge, документах и векторном
  хранилище;
- удалены временные ветки завершённого маршрута, `main` синхронизирован с
  `origin/main`.

Мы не меняли semantic JSON, правила Gemini master, извлечение изображений,
OCR или Markdown runtime. Проблема оказалась не в распознавании таблицы.

## Что теперь работает

Пользователь может загрузить документ через защищённый Broker Reports Action.
Gate 1 обрабатывает документ, сохраняет результат в ArtifactStore и создаёт
контекст для Gate 2.

Обе модели Gate 2 теперь продолжают работу в том же чате. Раньше после
переключения модели OpenWebUI создавал другой чат, из-за чего Gate 2 не видел
результат Gate 1.

Релиз проверен по точным хешам. Functions, Action, prompts и image остались
неизменными там, где изменение не требовалось.

## Почему работа пока не закрыта полностью

Остались две серверные проблемы.

### 1. Неполный охват отчёта

При обычном запросе из браузера domain-шаг Gate 2 по умолчанию берёт только
один документ, один source unit и один segment.

В контрольном отчёте найдено десять source units. Поэтому система способна
создать формально завершённый, но неполный answer context.

До публикации answer context необходимо либо обработать весь разрешённый
scope, либо явно завершить операцию безопасной блокировкой.

### 2. Answer context не передаётся отвечающей модели

Система умеет собрать и безопасно проверить answer context. Но в
производственном браузерном маршруте пока нет вызова, который передаёт этот
контекст отвечающей модели вместе с вопросом пользователя.

Сейчас domain Function возвращает только краткую сводку о завершении
обработки. Обычная модель видит историю чата, но не получает защищённый
answer context.

Ранее проверка трёх контрольных метрик прошла успешно через отдельный
технический harness. Это доказывает качество данных, но не доказывает
пользовательский browser workflow.

## Почему новый PDF-прогон не запускался

Без устранения этих двух проблем свежий прогон не мог выполнить обязательное
условие `answer-context-only`.

Повторная обработка PDF создала бы новые приватные артефакты и нагрузку, но не
закрыла бы acceptance. Поэтому прогон был остановлен на безопасном preflight.

## Что нужно сделать дальше

Нужны две отдельные небольшие runtime-коррекции:

1. обеспечить полный customer scope для domain Gate 2 или fail-closed при
   наличии необработанных units и segments;
2. добавить в поддерживаемую Function режим ответа, который по
   авторизованному chat scope получает готовый answer context и передаёт
   отвечающей модели только этот контекст и текущий вопрос.

Повторный вопрос должен использовать уже собранный context без повторной
обработки документа.

После исправлений потребуется один свежий браузерный прогон:

`PDF → Action → processing → completion → вопрос → ответ → follow-up`.
