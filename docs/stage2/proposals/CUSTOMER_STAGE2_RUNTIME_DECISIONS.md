# Решения после runtime-аудита OpenWebUI

Статус документа: справка для согласования с заказчиком. Не является финальным
техническим заданием.

`runtime-аудит` - это проверка работы системы на реальном стенде. В этом
документе простыми словами описано, что уже подтвердилось, какие решения нужны
со стороны заказчика и что можно делать после согласования.

## 1. Зачем нужен этот документ

Мы проверили, какие возможности OpenWebUI реально работают на стенде. Часть
возможностей подтвердилась для тестового сценария. По нескольким вопросам нужны
решения заказчика, потому что это не только технические настройки, но и правила
работы компании.

Эта справка дополняет основное предложение по Stage 2. Она не заменяет его и
не является техническим заданием.

## 2. Что подтвердил runtime-аудит

Простыми словами:

- рабочие сценарии можно собирать штатными средствами OpenWebUI: через группы,
  модели, шаблоны и базу знаний;
- доступ к отдельной модели, шаблону и базе знаний можно ограничивать по
  группе;
- транскрибация через Stage 2 sidecar остается правильным путем. Sidecar - это
  отдельный сервис рядом с OpenWebUI, который выполняет транскрибацию;
- Web Search технически работает. Web Search - это поиск в интернете из
  интерфейса AI-чата;
- сейчас Web Search доступен глобально, то есть не только отдельной группе;
- загрузка простых файлов работает на базовом уровне;
- штатная аналитика OpenWebUI доступна, но ее достаточность нужно подтвердить
  по ожиданиям заказчика.

Важно: это не означает, что все готово к промышленной эксплуатации. Корректные
формулировки: "подтверждено для тестового сценария", "требует решения
заказчика", "требует отдельной проверки".

## 3. Решение 1. Удаление чатов и хранение истории

Что показал аудит:

- сейчас пользователь может удалить чат;
- запрет удаления рабочих чатов не считается закрытым.

Почему это важно:

Если чат используется как рабочий инструмент, может понадобиться хранить
историю: для контроля, восстановления, аудита или разбора спорных ситуаций.

Варианты:

1. Оставить удаление разрешенным.
2. Запретить пользователям удалять рабочие чаты.
3. Разрешить удаление личных или черновых чатов, но хранить рабочие сценарии.
4. Ввести retention policy - правило, сколько времени хранить историю.

Что нужно решить заказчику:

- нужно ли запрещать удаление рабочих чатов;
- какие чаты считать рабочими;
- сколько хранить историю;
- кто имеет право удалять или архивировать чаты.

## 4. Решение 2. Web Search: всем или только группам

Что показал аудит:

- Web Search технически работает;
- сейчас он включен глобально;
- режим "только для отдельной группы" пока нельзя считать согласованным
  промышленным включением.

Почему это важно:

Поиск в интернете отправляет запрос наружу. В запросе могут случайно оказаться
внутренние данные, персональные данные или коммерческая информация.

Варианты:

1. Оставить Web Search доступным всем пользователям.
2. Включить Web Search только пилотной группе.
3. Разрешить Web Search всем, но с инструкцией и запрещенными типами запросов.
4. Оставить Web Search в тестовой эксплуатации до накопления статистики.

Что нужно решить заказчику:

- кому можно использовать Web Search;
- какие запросы запрещены;
- нужно ли показывать предупреждение пользователю;
- кто контролирует usage/cost signal и логи.

## 5. Решение 3. Видимость рабочих чатов руководителями

Что показал аудит:

- обычные группы и права доступа позволяют ограничивать модели, шаблоны и базу
  знаний;
- автоматическая видимость всех рабочих чатов руководителем не доказана;
- нельзя просто дать руководителю админские права и считать это решением.

Почему это важно:

Руководителю может быть нужно видеть рабочие результаты команды. При этом
нельзя случайно открыть личные, черновые или не относящиеся к работе чаты.

Варианты:

1. Руководитель не видит чаты сотрудников, только общие шаблоны и сценарии.
2. Руководитель видит только явно опубликованные или открытые для него рабочие
   чаты.
3. Руководитель получает отдельный отчет или экспорт по рабочим сценариям.
4. Делать отдельную доработку только если штатных возможностей не хватает.

Что нужно решить заказчику:

- должен ли руководитель видеть рабочие чаты;
- какие чаты считаются рабочими;
- кто видит личные или черновые чаты;
- нужен ли отчет вместо прямого доступа к чатам.

## 6. Решение 4. Аналитика и usage/cost visibility

Что показал аудит:

- штатная аналитика OpenWebUI доступна;
- видны агрегаты по пользователям и моделям;
- мгновенное появление тестовых пользователей в аналитике не подтвердилось;
- жесткий биллинг не является обязательным первым шагом.

`Аналитика` - это статистика использования: кто пользуется системой, какими
моделями и сколько примерно расходуется.

`Жесткий биллинг` - это когда система не только показывает расходы, но и
автоматически ограничивает их.

`Gateway` - это отдельный шлюз между OpenWebUI и AI-провайдерами. Он может быть
нужен для строгого учета и лимитов, но его не стоит внедрять без отдельного
решения.

Варианты:

1. На первом этапе достаточно штатной аналитики OpenWebUI.
2. Добавить ручную сверку с отчетами провайдеров.
3. Позже внедрить отдельный gateway для строгого учета и лимитов.
4. Не внедрять hard billing - жесткий учет и ограничение расходов, пока
   заказчик не подтвердит необходимость.

Что нужно решить заказчику:

- достаточно ли видеть статистику;
- нужны ли жесткие лимиты;
- кто отвечает за контроль расходов;
- нужна ли детализация по отделам или группам.

## 7. Что можно делать без дополнительных решений заказчика

Можно готовить:

- шаблоны рабочих сценариев;
- структуру каталога моделей;
- инструкции для администратора и пользователя;
- тестирование Web Search только в тестовом режиме;
- список запрещенных типов запросов;
- шаблоны политики данных;
- synthetic test data - искусственные тестовые файлы без данных заказчика.

## 8. Что нельзя делать без согласования

Нельзя:

- включать Web Search всем пользователям как промышленную функцию;
- запрещать или разрешать удаление рабочих чатов как финальную политику;
- открывать руководителям доступ к чатам сотрудников;
- отправлять реальные клиентские документы внешним провайдерам;
- подключать новых production-провайдеров, то есть провайдеров для рабочей
  эксплуатации;
- внедрять gateway или hard billing;
- делать глубокую доработку OpenWebUI.

## 9. Что должно получиться после согласования

После обсуждения заказчик должен выбрать:

- политику удаления и хранения чатов;
- правила Web Search;
- модель видимости рабочих чатов руководителями;
- подход к аналитике и usage/cost control.

После этого можно подготовить полноценное техническое задание и оценку работ.
