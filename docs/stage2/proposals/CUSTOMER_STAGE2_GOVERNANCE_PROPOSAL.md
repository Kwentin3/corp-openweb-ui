# Предложение по развитию корпоративного AI-чата. Stage 2

Статус документа: предварительное предложение для согласования направления.

Этот документ не является финальным техническим заданием. Он нужен, чтобы
заказчику было проще обсудить следующий этап не с чистого листа. После
согласования направления на его основе можно подготовить полноценное ТЗ.

## 1. Назначение документа

У нас уже есть рабочая база корпоративного AI-чата на OpenWebUI и первые
проверенные функции Stage 2. Следующий шаг важно выбрать не как отдельную
"кнопку", а как управляемый слой для корпоративного использования ИИ.

Цель этого документа:

- коротко зафиксировать текущую ситуацию;
- предложить следующий логичный этап;
- показать, что в него входит и что не входит;
- собрать вопросы, которые нужно согласовать с заказчиком.

## 2. Что уже сделано

На текущий момент закрыта базовая часть проекта:

- поднят и принят базовый корпоративный OpenWebUI для небольшой группы
  пользователей;
- подготовлена документационная и архитектурная база Stage 2;
- реализован и закрыт для текущего этапа STT MVP: загрузка аудио/видео,
  подготовка аудио в браузере, серверная отправка на STT-провайдера и возврат
  текста в интерфейс OpenWebUI;
- Web Search проверен на уровне provider baseline: подтверждено, что OpenWebUI
  может работать с выбранными поисковыми путями.

Важно: Web Search пока нельзя считать production-ready функцией для всех
пользователей. Он остается в тестовой эксплуатации: нужно накопить статистику,
проверить доступы, ограничения, логи, стоимость и правила использования.

## 3. Какая проблема появляется сейчас

Проект начинает переходить от "чата с ИИ" к корпоративному инструменту.
На этом этапе главные риски уже не только технические.

Появляются управленческие вопросы:

- кто из сотрудников какими моделями пользуется;
- какие данные можно отправлять внешним AI-провайдерам;
- какие данные нельзя отправлять никогда;
- какие сценарии разрешены для обычных пользователей;
- какие сценарии доступны только отдельным группам;
- как видеть использование и примерную стоимость;
- как не допустить хаотичного применения ИИ сотрудниками.

Если эти правила не описать заранее, система будет развиваться набором
разрозненных функций. Это повышает риски по данным, стоимости, доступам и
ожиданиям пользователей.

## 4. Наша рекомендация

Следующей фичей предлагаем сделать:

**Stage 2 Provider & Data Governance**.

Проще: **каталог провайдеров и моделей плюс политика данных по классам**.

Это не отдельная кнопка в интерфейсе. Это первый "пульт управления"
корпоративным использованием ИИ.

На практике это должна быть понятная таблица или набор правил, где видно:

- какие AI-модели и провайдеры доступны;
- для каких групп пользователей они доступны;
- для каких задач их можно использовать;
- с какими типами данных они разрешены;
- какие ограничения действуют для Web Search;
- где достаточно наблюдать стоимость, а где нужны жесткие ограничения;
- какие сценарии пока остаются пилотными.

Такой слой нужен до расширения документов, OCR, Excel, брокерских отчетов и
3-НДФЛ. Эти сценарии почти всегда связаны с чувствительными данными, поэтому
им сначала нужны правила.

## 5. Что войдет в следующий этап

Предварительный scope следующего этапа:

1. Описать классы данных.
   Например: публичные данные, внутренние рабочие данные, персональные данные,
   финансовые документы, клиентские данные, секреты и ключи.

2. Описать классы провайдеров и моделей.
   Например: внешние зарубежные провайдеры, российские провайдеры,
   self-hosted или внутренние компоненты, тестовые модели, production-модели.

3. Подготовить первичную матрицу разрешений.
   В ней должно быть видно, какие типы данных можно отправлять каким
   провайдерам и при каких условиях.

4. Проверить группы и роли OpenWebUI.
   Нужно понять, как штатными средствами OpenWebUI ограничивать доступ к
   моделям, сценариям и Web Search.

5. Проверить базовую видимость использования и стоимости.
   На первом этапе может быть достаточно наблюдаемости: кто пользуется, какими
   моделями, примерно с какой стоимостью. Жесткий billing/gateway не нужно
   внедрять без отдельного решения.

6. Зафиксировать правила тестовой эксплуатации Web Search.
   Нужно определить, кому он доступен, какие запросы запрещены, как проверяются
   источники, что логируется и как контролируется стоимость.

7. Подготовить список решений для заказчика.
   Этот список должен стать основой полноценного ТЗ на следующий этап.

## 6. Что не входит в этот этап

Чтобы этап остался управляемым, предлагаем явно исключить:

- глубокий fork OpenWebUI;
- внедрение тяжелого billing/gateway без отдельного решения;
- production-документооборот, OCR и Excel без реальных клиентских примеров;
- полноценный rollout Web Search для всех пользователей без статистики и
  согласованных правил;
- финальную политику безопасности без подтверждения заказчика;
- автоматическую отправку любых чувствительных данных внешним провайдерам;
- обещание, что любой документ или таблица будут корректно обработаны ИИ.

Если в ходе согласования выяснится, что часть этих пунктов нужна уже сейчас,
их лучше оформить отдельным решением и отдельным scope.

## 7. Что нужно согласовать с заказчиком

Для подготовки полноценного ТЗ нужны ответы на вопросы:

1. Какие группы сотрудников первыми входят в следующий этап?
2. Какие типы данных считаются чувствительными?
3. Какие данные нельзя отправлять внешним провайдерам ни при каких условиях?
4. Какие модели и провайдеры заказчик готов использовать?
5. Есть ли предпочтение по российским, зарубежным или self-hosted путям?
6. Нужна ли строгая бюджетная отсечка или пока достаточно наблюдаемости?
7. Какие сценарии важнее всего: Web Search, документы, OCR, Excel, брокерские
   отчеты, 3-НДФЛ, рабочие шаблоны, аналитика использования?
8. Какие примеры документов заказчик сможет предоставить позднее для проверки?
9. Кто со стороны заказчика принимает решения по данным, доступам и стоимости?
10. Какие действия пользователей нужно запретить сразу?

## 8. Ожидаемый результат согласования

После обсуждения этого документа должны появиться:

- выбранный состав следующей фичи;
- границы этапа;
- список исключений;
- решения по классам данных;
- решения по провайдерам и моделям;
- понимание, какие группы пользователей участвуют первыми;
- правила тестовой эксплуатации Web Search;
- основа для полноценного технического задания.

Хороший результат согласования - не большой список пожеланий, а короткий набор
решений: что делаем сейчас, что не делаем сейчас и какие данные можно
использовать.
