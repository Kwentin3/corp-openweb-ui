# Stage 2: объем работ и вопросы для согласования

Статус: проект для согласования с заказчиком.
Дата: 2026-06-30.

Этот документ нужен для согласования состава Stage 2 перед подготовкой договора
и приложения к нему. Это не договор, не акт, не счет и не технический аудит.

Денежные условия в этом документе не фиксируются. Здесь указаны только
ориентировочные трудозатраты в часах.

## 1. Короткое резюме

PRD-0 уже выполнен. Stage 2 - следующий шаг развития корпоративного OpenWebUI:
из обычного чата с ИИ система постепенно становится рабочим инструментом для
сотрудников.

Текущий Stage 2 не закрывает весь большой PRD-1. Предлагается первый
практический состав работ, который можно использовать в пилоте:

- базовая транскрибация аудио и видео внутри OpenWebUI;
- обработка расшифровок по готовым шаблонам;
- базовый Web Search, то есть поиск актуальной информации через интернет
  внутри чата;
- набор корпоративных шаблонов для типовых рабочих задач;
- короткая инструкция для пользователей;
- базовая пилотная матрица доступов;
- документация и критерии приемки Stage 2.

## 2. Почему предлагается именно такой состав Stage 2

У заказчика уже есть отдельный сервис аудиотранскрибации. Логичный следующий
шаг - постепенно переносить эту ценность в OpenWebUI, чтобы сотрудник работал
в одной системе: создал чат под клиента или задачу, загрузил аудио, получил
расшифровку, сделал пересказ, протокол, список задач, письмо по итогам и при
необходимости нашел актуальные источники через Web Search.

Мы не предлагаем набор случайных функций. Мы предлагаем первый рабочий слой
корпоративной ИИ-платформы: то, чем сотрудники смогут пользоваться в реальных
рабочих сценариях.

Подход также снижает количество разрозненных сервисов. Там, где это возможно,
используются штатные возможности OpenWebUI, чтобы не усложнять поддержку и не
делать глубокую переработку продукта без отдельного решения.

## 3. Что уже реализовано

### Базовая транскрибация аудио и видео

Уже подготовлен базовый сценарий:

- пользователь загружает аудио или видео в OpenWebUI;
- запускает транскрибацию;
- получает текст расшифровки в чате;
- история остается внутри обычного чата OpenWebUI, если пользователь не удаляет
  этот чат;
- отдельный раздел "Встречи" не создается.

Транскрибация реализована так, чтобы пользователь оставался внутри OpenWebUI, а
ключи внешнего сервиса транскрибации не попадали в браузер пользователя.

### Базовый Web Search

Уже подготовлена базовая возможность поиска актуальной информации через
интернет внутри OpenWebUI. Это рабочая основа для пилота, но не полная система
сложных правил, журналов и ограничений.

### Документационная и приемочная база

Подготовлены материалы Stage 2: архитектурная рамка, backlog, ограничения,
приемочная матрица и разделение текущего состава работ от будущих направлений.

## 4. Что предлагается включить в текущий Stage 2

### F1. Базовая транскрибация аудио и видео в OpenWebUI

Статус: уже реализовано.

Что получит заказчик:

- загрузку аудио или видео в чат;
- запуск транскрибации из OpenWebUI;
- получение текста расшифровки в этом же рабочем контуре.

Оценка трудозатрат: уже выполнено.

Ограничение: это не отдельный продукт для встреч и не сложный календарный
процесс.

### F2. Обработка расшифровок по шаблонам

Статус: к реализации.

Что получит заказчик:

- после расшифровки пользователь сможет выбрать готовое действие;
- система обработает текст по выбранному шаблону;
- результат вернется в этот же чат.

Примеры действий: краткий пересказ, подробный пересказ, протокол встречи,
список задач, решения, открытые вопросы, итоги для руководителя, письмо по
итогам.

Оценка трудозатрат: 16-32 ч.

Ограничение: результат остается AI-подготовкой и требует проверки человеком.

### F3. Стартовый набор шаблонов для расшифровок

Статус: к реализации.

Что получит заказчик:

- стартовый набор понятных шаблонов;
- единый стиль результатов;
- возможность согласовать, какие шаблоны обязательны для первого пилота.

Оценка трудозатрат: 8-16 ч.

Ограничение: это стартовый набор, а не библиотека под все будущие процессы.

### F4. Управление шаблонами через OpenWebUI

Статус: к реализации с проверкой на целевом стенде.

Что получит заказчик:

- настройку общих шаблонов в OpenWebUI;
- разделение общих и личных шаблонов, если это подтверждается на целевом
  стенде;
- понятное правило, кто может редактировать общие шаблоны.

Оценка трудозатрат: 8-16 ч.

Ограничение: отдельный сложный редактор шаблонов не входит в текущий этап.

### F5. Учет говорящих в расшифровке

Статус: к реализации.

Что получит заказчик:

- более удобную обработку расшифровок, где есть разделение по говорящим;
- подготовку протоколов, задач и решений с учетом реплик участников, если эти
  данные доступны в исходной расшифровке.

Оценка трудозатрат: 8-16 ч.

Ограничение: система не обещает безошибочно определять реальные имена
участников. При необходимости имена уточняются вручную.

### F6. Простой экспорт результата в DOCX

Статус: к реализации.

Что получит заказчик:

- выгрузку обработанного результата в файл Microsoft Word;
- документ для дальнейшей ручной правки;
- базовую структуру текста без сложного фирменного оформления.

Оценка трудозатрат: 8-16 ч.

Ограничение: PDF, сложные шаблоны Word и фирменное оформление выносятся в
будущие этапы.

### F7. Базовый Web Search

Статус: уже реализовано как базовая возможность.

Что получит заказчик:

- возможность искать актуальную информацию через интернет внутри OpenWebUI;
- основу для рабочих сценариев с источниками;
- базу для дальнейшего согласования правил использования.

Оценка трудозатрат: уже выполнено.

Ограничение: полная система правил, журналов, лимитов и детального управления
Web Search не входит в текущий этап.

### F8. Готовые сценарии Web Search

Статус: к реализации.

Что получит заказчик:

- сценарий "найти актуальную информацию";
- сценарий "сравнить варианты";
- сценарий "проверить утверждение через интернет";
- сценарий "найти официальные источники";
- сценарий "подготовить краткую справку с источниками".

Оценка трудозатрат: 4-8 ч.

Ограничение: сценарии помогают пользоваться поиском, но не заменяют
юридическую, финансовую или экспертную проверку.

### F9. Общий корпоративный набор шаблонов

Статус: к реализации.

Что получит заказчик:

- шаблон "сделать краткое резюме текста";

- шаблон "сделать список задач";

Оценка трудозатрат: 8-16 ч.

Ограничение: это стартовый рабочий набор, а не полная методология всех

### F10. Короткая инструкция для пользователей

Статус: к реализации.

Что получит заказчик:

- как создать чат под клиента или задачу;
- как пользоваться транскрибацией;

Ограничение: это короткая пилотная инструкция, не полный учебный курс.

### F11. Базовая пилотная матрица доступов

Статус: к реализации.

## 6. Что не входит в текущий Stage 2

Эти направления не отменяются, но относятся к следующим этапам и требуют
отдельного согласования:

- брокерские отчеты / 3-НДФЛ;
- OCR / VL OCR, то есть распознавание сканов и сложных документов;
- полноценная работа с PDF/DOCX/XLSX;
- интеграция с CRM и системами задач;
- автоматическая постановка задач во внешних системах;
- отдельный раздел "Встречи";
- сложный процесс ведения встреч;
- экспорт в PDF;
- фирменное оформление DOCX;
- доступ руководителей к рабочим чатам;
- хранение, запрет удаления и аудит;
- корпоративная авторизация AD/SSO/SCIM;
- полноценная аналитика расходов;
- полное управление правилами Web Search;
- маскирование и токенизация данных.

## 7. Вопросы для согласования с заказчиком

### По транскрибации и шаблонам

1. Какие шаблоны обработки расшифровки нужны в первую очередь?
2. Достаточно ли стартового набора: краткий пересказ, подробный пересказ,
   протокол, задачи, решения, вопросы без ответа, итоги для руководителя,
   письмо по итогам?
3. Нужно ли вручную указывать имена участников после транскрибации?
4. Где должен появляться результат: отдельным сообщением в этом же чате или в
   поле ввода для ручной правки перед отправкой?

### По DOCX

5. Достаточно ли простого DOCX без фирменного оформления?
6. Нужно ли добавлять в DOCX название чата, дату и тип шаблона?
7. Нужны ли отдельные красивые DOCX-шаблоны позже?

### По Web Search

8. Кому включаем Web Search: всем пользователям или пилотной группе?
9. Какие сценарии поиска самые важные?
10. Нужно ли требовать ссылки на источники в ответе?
11. Есть ли темы, по которым поиск запрещен или нежелателен?

### По шаблонам

12. Кто должен редактировать общие шаблоны?
13. Можно ли пользователям создавать личные шаблоны?
14. Нужно ли разрешать пользователям делиться шаблонами?

### По доступам

15. Кто входит в пилотную группу?
16. Кто является администратором?
17. Кому доступна транскрибация?
18. Кому доступен Web Search?
19. Кто может управлять шаблонами?

### По приемке

20. Кто со стороны заказчика принимает результат?
21. Какие 2-3 тестовых сценария считаем достаточными для приемки?
22. Какие тестовые аудиофайлы можно использовать?
23. Нужно ли проводить демонстрацию перед подписанием акта?

### Дополнительные вопросы для более точного согласования

24. Какие три шаблона считаем обязательными для первого запуска?
25. Нужно ли показывать пользователю предупреждение перед отправкой
    чувствительных данных в чат или Web Search?
26. Кто будет поддерживать шаблоны после пилота?
27. Как называем пользовательские действия в интерфейсе: "Расшифровать",
    "Сделать протокол", "Подготовить письмо", "Найти источники"?
28. Какие результаты должны быть показаны на демонстрации: аудиофайл,
    обработанная расшифровка, DOCX, Web Search и матрица доступов?

## 8. Следующий шаг после согласования

После согласования этого документа можно подготовить договор и приложение к
нему. В приложение будет перенесен согласованный перечень результатов,
ограничения и критерии приемки.

Практично вынести в приложение:

- список функций F1-F12 с согласованными статусами;
- список того, что не входит в текущий этап;
- критерии приемки по 2-3 пользовательским сценариям;
- правила по шаблонам, Web Search и пилотным доступам;
- перечень вопросов, по которым заказчик дал решение.
