# OpenWebUI Corporate Chat Stage 2 Summary

Статус: customer-facing summary
Источник: [OPENWEBUI_CORPORATE_CHAT_PRD_1_REFINED.md](OPENWEBUI_CORPORATE_CHAT_PRD_1_REFINED.md)
Changelog: [OPENWEBUI_CORPORATE_CHAT_PRD_1_REFINED_CHANGELOG.md](OPENWEBUI_CORPORATE_CHAT_PRD_1_REFINED_CHANGELOG.md)
Назначение: согласование Stage 2, объема работ и предварительной оценки трудозатрат

## 1. Коротко

PRD-1 - это не "подключить еще модели", а развитие текущего корпоративного AI-чата в управляемую среду рабочих сценариев.

Главная ценность Stage 2:

- сотрудники работают не в пустом чате, а в подготовленных сценариях;
- есть группы, шаблоны, prompts, инструкции и ограничения;
- появляются понятные сценарии для документов, транскрибации, web-search и брокерских отчетов;
- расходы и модели становятся хотя бы базово видимыми и управляемыми;
- сложные вещи не включаются автоматически, а оформляются отдельными решениями.

## 2. Что входит в базовый Stage 2

| Блок | Что входит | Результат для заказчика |
| ---- | ---------- | ----------------------- |
| Рабочие сценарии | Минимум 3 повторяемых сценария с правилами и ожидаемыми результатами | Сотрудники получают не общий пустой чат, а понятные рабочие маршруты |
| Группы доступа | Роли, группы, доступ к сценариям и моделям | Разные сотрудники видят только разрешенные возможности |
| Curated workspace models | Ограниченный каталог рабочих моделей вместо хаотичного списка provider-моделей | Модели проще выбирать и контролировать |
| Shared prompts/templates | Общие prompts, шаблоны и команды для типовых задач | Опыт сильных пользователей можно переиспользовать |
| Knowledge/instructions | Общие инструкции, методички и базовый knowledge context | Сценарии работают по единым правилам |
| Model catalog | Назначение моделей, ограничения, рекомендуемые сценарии применения | Понятно, какую модель для чего использовать |
| Price catalog | Каталог тарифных единиц: LLM tokens, web-search requests, STT hours, storage/files | Расходы можно считать и обсуждать предметно |
| Basic usage analytics | Проверка OpenWebUI Analytics, usage по пользователям/моделям или documented decision | Администратор получает базовую видимость использования |
| Policy по допустимым данным | Правила, что можно и нельзя отправлять в модели и внешние сервисы | Снижается риск утечки чувствительной информации |
| User/admin instructions | Короткие инструкции для сотрудников и администратора | Этап можно передать в эксплуатацию без устных договоренностей |
| Smoke/acceptance checks | Проверки сценариев, доступа, провайдеров, документов и web-search | Приемка опирается на проверяемый список, а не на впечатления |

## 3. Пилотные сценарии

| Сценарий | Что проверяем | Ожидаемый результат | Ограничения |
| -------- | ------------- | ------------------- | ----------- |
| Брокерские отчеты / 3-НДФЛ | Загрузка отчета, шаблон разбора, предупреждения, ручная проверка | Черновая аналитическая помощь для подготовки разбора | Это не налоговая консультация, не замена специалиста и не автоматическая подача декларации |
| Транскрибация встреч | Native STT/OpenAI-compatible путь, Lemonfox, mp3, права доступа | Сотрудник получает текстовую расшифровку и может использовать ее в сценарии | Видео, большие файлы, diarization и browser-side conversion требуют отдельной проверки |
| Работа с PDF/DOCX/XLSX | Простые документы, извлечение текста/структуры, ограничения по сложным файлам | Понятный pilot для анализа простых офисных файлов | Не обещается корректная обработка любых Word/PDF/Excel |
| Web-search | Провайдер, права доступа, стоимость request, качество ответа | Управляемый web-search для разрешенных ролей/сценариев | Не включать всем автоматически без политики данных и стоимости |
| Provider evaluation | DeepSeek, YandexGPT, GigaChat, Claude: подключение, качество, цена, ограничения | Рекомендация: production / pilot / reject / deferred | Доступы, ключи, квоты и балансы предоставляет оператор/заказчик |

## 4. Что не входит в базовый объем

Эти пункты учтены в PRD-1, но не входят в базовый Stage 2 автоматически. Если они нужны, по каждому готовится отдельное решение, оценка и acceptance criteria.

- fork OpenWebUI;
- browser-side ffmpeg.wasm;
- LiteLLM/gateway;
- hard budgets/enforcement;
- full AD lifecycle / SCIM rollout;
- production-grade document pipeline;
- сложный Excel parser;
- OCR/layout-aware PDF;
- production генерация DOCX/XLSX;
- доступ руководителей к чатам сотрудников;
- полноценный RAG по всем документам компании;
- агентная система с действиями в 1С/CRM.

## 5. Почему офисные документы оцениваются осторожно

Офисные документы нельзя считать простой настройкой чат-бота.

- DOCX - это не просто текст, а структура, стили, таблицы, комментарии, track changes и вложения.
- PDF часто является визуальной страницей, а не логическим документом.
- XLSX - это рабочая книга с листами, формулами, типами данных, скрытыми строками, merged cells, сводными таблицами и внешними ссылками.
- LLM хорошо помогает объяснять и анализировать, но точные расчеты и преобразования должны выполнять инструменты.
- Для PRD-1 целевой подход: простые документы, честные ограничения и отдельный parser/tool decision для сложных кейсов.

## 6. Предварительная оценка трудозатрат

Оценка дана в часах. Она не включает стоимость LLM/API, web-search, STT, сервера, хранения и подписок. После discovery диапазоны нужно уточнить по фактической версии OpenWebUI, доступам, провайдерам и требованиям заказчика.

| Блок | Тип работ | Оценка часов, min | Оценка часов, expected | Оценка часов, max | Допущения | Риски |
| ---- | --------- | ----------------: | ---------------------: | ----------------: | --------- | ----- |
| Discovery и проверка текущего состояния | Аудит repo/runtime facts, подтверждение PRD-0 baseline, final scope | 4 | 6 | 8 | Есть доступ к repo и оператору | Runtime-факты могут отличаться от документации |
| Проверка версии OpenWebUI и native capabilities | Версия, feature parity, Groups, Analytics, Knowledge, STT, web-search | 3 | 5 | 8 | Версия доступна для проверки | Возможен pinned OpenWebUI без нужных возможностей |

| Shared prompts/templates/knowledge | Шаблоны, prompts, инструкции, базовый knowledge context | 8 | 12 | 18 | Заказчик дает примеры задач и материалов | Без владельца шаблонов качество будет слабее |

## 7. Итоговая оценка

| Вариант | Что входит | Оценка часов | Комментарий |
| ------- | ---------- | ------------: | ----------- |
| Base Stage 2 Lite | Обязательное ядро, 3 рабочих сценария, 2-3 pilot checks, базовая видимость расходов, provider evaluation light | 72-108 | Хороший baseline для старта без тяжелых enterprise-срезов |
| Practical Stage 2 | Ядро, 3-4 пилотных сценария, более полный provider/web-search/document/STT контур, AD/SSO discovery | 112-164 | Наиболее реалистичный вариант для согласования полноценного Stage 2 |
| Extended Stage 2 | Ядро, расширенные slices, больше тестовых наборов и handoff, но без full enterprise-платформы | 168-240 | Подходит, если заказчик хочет шире проверить документы, STT, web-search и провайдеров |

Optional slices из раздела выше добавляются отдельно только после отдельного решения.

## 8. Допущения оценки

- доступ к репозиторию есть;
- доступ к серверу/оператору может потребоваться для runtime checks;
- секреты не печатать и не переносить в документы;
- API keys и балансы предоставляет оператор/заказчик;
- AD/SSO зависит от инфраструктуры заказчика;
- тарифы провайдеров требуют повторной проверки перед коммерческим предложением;
- OpenWebUI может быть pinned на версию PRD-0, поэтому feature parity нужно проверить;
- сложные документы требуют отдельного parser/tool decision;
- browser-side ffmpeg.wasm не считается обязательным без proof;
- hard billing не считается обязательным без gateway decision.

## 9. Acceptance summary

Результатом базового Stage 2 считается:

- минимум 3 рабочих сценария;
- группы/доступы;
- prompts/templates;
- model catalog;
- price catalog;
- basic usage analytics или documented decision;
- один STT/transcription pilot;
- один document pilot;
- один web-search pilot или documented defer;
- provider evaluation;
- user/admin docs;
- acceptance checklist;
- обновленный README.

## 10. Решения заказчика перед стартом

Перед стартом нужно согласовать:

- какие 3 сценария приоритетны;
- кто AI-методолог/владелец шаблонов;
- какие группы пользователей участвуют;
- какие данные разрешено отправлять во внешние модели;
- нужен ли web-search всем или только отдельным ролям;
- нужна ли транскрибация аудио только mp3 или сразу видео/большие файлы;
- нужен ли AD/SSO в этом этапе или только discovery;
- достаточно ли basic analytics или нужен hard billing/gateway;
- какие документы и шаблоны заказчик предоставит для тестов;
- какие провайдеры реально нужно включать в production, а какие только исследовать.
