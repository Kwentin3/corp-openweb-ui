# PRD-0: Фаза смотрин корпоративного LLM-чата на OpenWebUI

Версия: v0.2
Статус: draft
Дата: 2026-06-09

## 1. Суть документа

Документ описывает нулевую фазу проекта: минимальный запуск OpenWebUI как единой точки входа в LLM-чат для ограниченной группы из 3-4 пользователей.

PRD-0 не описывает полноценную AI-платформу и не является финальным MVP. Его задача - быстро проверить, появится ли у сотрудников реальная потребность в едином корпоративном LLM-чате.

## 2. Цель фазы

Развернуть OpenWebUI на домене `gpt.alpha-soft.ru`, подключить два API-провайдера - OpenAI и Gemini, создать 3-4 пользовательских аккаунта и дать участникам возможность задавать вопросы в чате.

Ключевая проверка: будут ли пользователи регулярно заходить в систему, задавать реальные рабочие вопросы, получать полезные ответы хотя бы от одного подключенного провайдера и хотеть пользоваться этим дальше.

## 3. Участники и роли

В фазе смотрин участвуют 3-4 человека:

- инициатор проекта;
- технический администратор;
- руководитель отдела;
- при необходимости пилотный пользователь.

На этом этапе нужны только две роли:

- администратор;
- пользователь.

Администратор создает пользователей, подключает API-ключи провайдеров, проверяет работоспособность системы и помогает участникам начать пользоваться чатом.

## 4. Объем PRD-0

Входит:

- установка OpenWebUI;
- публикация OpenWebUI на домене `gpt.alpha-soft.ru`;
- настройка Traefik и HTTPS;
- подключение OpenAI и Gemini средствами OpenWebUI;
- OpenAI primary provider через server-local `.env`;
- Gemini secondary provider через OpenWebUI Admin UI;
- точные model id: OpenAI `gpt-5.4-mini`, Gemini `gemini-3.5-flash`;
- создание 3-4 пользовательских аккаунтов;
- отключение или ограничение самостоятельной публичной регистрации;
- проверка отправки запроса к модели и получения ответа;
- проверка сохранения истории чатов;
- дешевая штатная кастомизация через env/Admin UI: мягкое имя инстанса, предупреждающий баннер, закрытая регистрация и базовые настройки доступа;
- базовый host hardening: UFW firewall и fail2ban для SSH;
- минимальное резервное копирование конфигурации и persistent volume OpenWebUI с `BACKUP_RETENTION_DAYS=7` по умолчанию;
- короткий runbook для эксплуатации;
- короткая пользовательская инструкция.

## 5. Вне объема PRD-0

Не входят:

- LiteLLM;
- model gateway;
- маршрутизация между провайдерами, бюджеты и per-user лимиты;
- выбор пользователем большого каталога моделей как отдельная продуктовая функция;
- web-поиск;
- корпоративный дашборд;
- РО-доступ к чатам подчиненных;
- скиллы Word, PDF, Excel;
- транскрибация;
- сложный RAG;
- memory layer;
- интеграции с 1С, CRM и другими корпоративными системами;
- white-label, смена бренда, логотипа или переупаковка OpenWebUI под собственный продукт;
- fork OpenWebUI, изменение frontend-кода, кастомный frontend, плагины, tools/functions;
- WAF, SIEM, DLP и корпоративный security audit.

Работа с файлами и документами не является обязательной частью PRD-0. Если штатная загрузка файлов в OpenWebUI работает без отдельной настройки, пользователи могут попробовать ее в формате best effort, но качество работы с документами не считается критерием готовности.

## 6. Целевая архитектура

Минимальная схема:

```text
пользователь -> gpt.alpha-soft.ru -> Traefik -> OpenWebUI -> OpenAI API
                                                   -> Gemini OpenAI-compatible API
```

OpenWebUI разворачивается на существующем или новом VPS через Docker/Docker Compose. Traefik используется для маршрутизации домена, выпуска и обслуживания HTTPS, а также проксирования запросов к контейнеру OpenWebUI.

Self-hosted OpenWebUI не означает, что модель работает локально. Запросы пользователей уходят внешним API-провайдерам.

Подключения к OpenAI и Gemini заводятся штатными средствами OpenWebUI.

## 7. Требования к доступу

- Доступ к OpenWebUI должен работать только через HTTPS.
- Публичные порты контейнера OpenWebUI не должны открываться напрямую наружу.
- На public perimeter нужны только `22/tcp`, `80/tcp` и `443/tcp`.
- UFW должен быть включен с default deny incoming и default allow outgoing.
- Fail2ban должен быть установлен, активен и защищать SSH через `sshd` jail.
- Самостоятельная публичная регистрация пользователей должна быть отключена или ограничена.
- Пользовательские аккаунты создаются администратором вручную.
- API-ключи хранятся только на сервере или в OpenWebUI Admin UI и не передаются пользователям.
- `WEBUI_SECRET_KEY` должен быть стабильным server-local секретом и не должен меняться между перезапусками или пересозданием контейнера.
- `CORS_ALLOW_ORIGIN` должен ограничивать browser/WebSocket origin доменом PRD-0.

## 8. Требования к провайдерам и моделям

В PRD-0 допускаются два API-провайдера:

- OpenAI: `https://api.openai.com/v1`;
- Gemini OpenAI compatibility: `https://generativelanguage.googleapis.com/v1beta/openai`.

Primary provider задается через `.env` переменными `OPENAI_API_BASE_URL`, `OPENAI_API_KEY` и опционально `DEFAULT_MODELS`.

Secondary provider добавляется вручную через OpenWebUI Admin UI в разделе Connections. Такой путь выбран консервативно: OpenWebUI поддерживает env-переменные для OpenAI-compatible API, но часть настроек является persistent config, а multi-provider env увеличивает риск ошибки порядка URL/key и сложнее проверяется в PRD-0.

Текущий deployment decision:

- OpenAI primary: `OPENAI_API_BASE_URL=https://api.openai.com/v1`, `DEFAULT_MODELS=gpt-5.4-mini`;
- Gemini secondary через Admin UI: `https://generativelanguage.googleapis.com/v1beta/openai`, Model IDs `gemini-3.5-flash`.

Если оператор намеренно тестирует именно Gemini 3 Flash Preview, точный model id - `gemini-3-flash-preview`. Строка `gemini-3-flash` без `preview` не используется как exact model code.

Перед пилотом оператор должен проверить:

- наличие OpenAI API key;
- наличие Gemini API key;
- billing/quota/region для каждого провайдера;
- доступ OpenAI API key к `gpt-5.4-mini`;
- доступ Gemini API key к `gemini-3.5-flash`;
- что хотя бы один provider дает ответ в чате.

Выбор нескольких моделей пользователем не является обязательным для PRD-0.

## 9. Требования к штатной кастомизации

В PRD-0 допускается только дешевая штатная кастомизация OpenWebUI через env и Admin UI:

- мягкое имя инстанса `Alpha Soft AI Chat` через `WEBUI_NAME`;
- warning banner через `WEBUI_BANNERS`;
- закрытая регистрация;
- базовые настройки доступа.

Это не white-label и не смена бренда OpenWebUI. Не входят fork, изменение логотипа, изменение frontend-кода, кастомный frontend, плагины, tools/functions и document skills.

Если `WEBUI_NAME` или `WEBUI_BANNERS` не применяются через env из-за persistent config, оператор настраивает их через Admin UI. Fork или патч OpenWebUI не допускается.

## 10. Требования к истории чатов

Система должна сохранять историю чатов пользователей.

Минимальная проверка:

1. Пользователь задает вопрос модели.
2. Ответ модели отображается в интерфейсе.
3. Чат остается доступен после выхода и повторного входа пользователя.
4. История не пропадает после перезапуска контейнера OpenWebUI.

## 11. Требования к безопасности

Пользователям запрещается загружать в чат:

- пароли;
- токены;
- API-ключи;
- приватные SSH-ключи;
- OAuth secrets;
- другие технические секреты и чувствительные доступы.

В пользовательской инструкции должно быть явно указано, какие данные нельзя отправлять в чат.

OpenWebUI должен показывать предупреждающий баннер с запретом отправлять пароли, токены, API-ключи, приватные SSH-ключи и закрытые персональные данные. Баннер не заменяет пользовательскую инструкцию и политику безопасности.

После стабилизации пилота администратор может сменить первичный пароль через UI и убрать bootstrap password из активного server-local `.env`, но это не blocker PRD-0.

## 12. Эксплуатационная документация

Для PRD-0 нужен короткий runbook, в котором указано:

- где расположен `docker-compose`;
- где расположен `env`-файл;
- как выполнить host hardening;
- как подключить OpenAI и Gemini;
- как проверить имя инстанса и предупреждающий баннер;
- как перезапустить сервис;
- как создать пользователя;
- как проверить логи;
- как проверить, что модель отвечает;
- как восстановить persistent volume из резервной копии.

Также нужна короткая пользовательская инструкция:

- как войти;
- как задать вопрос;
- какие данные нельзя загружать;
- к кому обращаться при проблемах.

## 13. Критерии готовности

PRD-0 считается готовым, если:

- домен `gpt.alpha-soft.ru` открывается по HTTPS;
- OpenWebUI работает;
- строгая TLS-проверка проходит без `curl -k`;
- UFW включен;
- наружу разрешены только необходимые публичные порты `22/tcp`, `80/tcp`, `443/tcp`;
- fail2ban установлен и активен;
- `sshd` jail активен;
- администратор может войти в систему;
- созданы 3-4 пользовательских аккаунта;
- пользователь может войти в систему;
- в интерфейсе отображается имя `Alpha Soft AI Chat` или подтвержденное оператором имя;
- пользователь видит предупреждающий баннер;
- баннер запрещает отправлять пароли, токены, API-ключи, приватные SSH-ключи и закрытые персональные данные;
- OpenAI primary provider с `gpt-5.4-mini` подключен или готов к подключению по runbook;
- Gemini secondary provider с `gemini-3.5-flash` подключен через Admin UI или готов к подключению по runbook;
- пользователь может задать вопрос модели;
- ответ хотя бы от одного provider приходит и отображается в чате;
- второй provider проверен или явно отмечен pending только из-за API key/quota/billing/region;
- история чата сохраняется после выхода пользователя;
- история чата сохраняется после перезапуска контейнера;
- API-ключи не передаются пользователям;
- `WEBUI_SECRET_KEY` задан в server-local `.env`;
- `CORS_ALLOW_ORIGIN` задан для HTTPS-домена PRD-0;
- подготовлены короткий runbook и пользовательская инструкция.

## 14. Критерии успеха через 1-2 недели

Фаза считается успешной, если участники:

- реально заходят в систему повторно;
- задают рабочие, а не только тестовые вопросы;
- получают ответы, которые считают полезными;
- могут назвать хотя бы 2-3 рабочих сценария, где такой чат им пригодится;
- хотят продолжить пользоваться единым LLM-чатом после смотрин.

## 15. Критерии неуспеха

Фаза считается неуспешной, если участники:

- не заходят повторно;
- задают только тестовые вопросы из любопытства;
- не находят рабочих сценариев;
- считают интерфейс бесполезным;
- не видят причин использовать единый чат вместо текущих инструментов.

## 16. Решение после фазы

После PRD-0 нужно принять одно из решений:

- развивать OpenWebUI дальше;
- расширять набор моделей и правила их использования;
- отдельно оценить функции, которые сейчас вне PRD-0;
- остановить проект.

Любое расширение за пределы штатной env/Admin UI кастомизации оформляется отдельным PRD, не как часть PRD-0 deployment path.

## 17. Оценка трудозатрат и финансовые условия

Ориентировочная трудоемкость PRD-0: 25-40 часов при условии, что:

- сервер доступен;
- домен управляем;
- Traefik можно настроить без нестандартных ограничений;
- API-ключи OpenAI и Gemini готовы или оперативно выдаются оператором.

Финансовые условия не фиксируются в GitHub-документации. В репозитории остается только оценка трудозатрат и scope boundary.

## 18. Главный риск

Главный риск - начать добавлять в нулевую фазу функции полноценного AI-портала и снова раздуть состав работ.

В PRD-0 делаем только одно: даем 3-4 людям рабочую точку входа в LLM-чат и смотрим, появится ли реальная потребность.

## 19. Открытые вопросы

- Какой VPS используется: существующий или новый?
- Кто будет администратором системы после запуска?
- Подтверждает ли оператор имя инстанса `Alpha Soft AI Chat`?
- Есть ли готовые OpenAI и Gemini API keys?
- Проверены ли billing/quota/region для обоих провайдеров?
- Кто собирает обратную связь от пользователей через 1-2 недели?

## 20. Короткий вывод

PRD-0 - это аккуратная фаза смотрин, а не финальный продукт: минимальный OpenWebUI на домене, OpenAI и Gemini без отдельного gateway, мягкое имя инстанса, предупреждающий баннер, несколько пользователей, базовый host hardening и проверка фактического интереса.
