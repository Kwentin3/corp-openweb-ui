# Демонстрационный стенд Hermes Corporate v1

## 1. Цель стенда

Мы предлагаем развернуть демонстрационный стенд Hermes Corporate v1: три независимые среды Hermes на одном новом VPS-сервере.

Пользователи смогут попробовать Hermes в отдельных изолированных средах. По итогам тестирования можно будет собрать обратную связь и сформировать требования к будущей полноценной корпоративной версии Hermes.

## 2. Что будет развернуто

Стенд будет состоять из одного нового сервера и трех отдельных Hermes-сред.

Каждая среда будет:

- открываться по своему адресу;
- иметь отдельную страницу входа;
- хранить свои данные отдельно;
- использовать свои настройки и журналы работы;
- работать независимо от двух других сред.

Пример адресов:

- `node1.example.com`
- `node2.example.com`
- `node3.example.com`

`example.com` здесь используется как пример. Реальный домен нужно согласовать отдельно.

## 3. Как это будет работать

Пользователь открывает ссылку на нужную Hermes-среду.

Система показывает страницу входа. После ввода логина и пароля пользователь попадает в свою среду Hermes.

Работа в одной среде не должна влиять на остальные. Если одна среда остановлена или перезапускается, две другие должны продолжать работать.

## 4. Что потребуется от заказчика

Для подготовки стенда нужно согласовать и предоставить:

- VPS-сервер;
- домен для стенда;
- доступ к DNS или ответственного, который создаст DNS-записи;
- SSH-доступ к серверу для настройки;
- список пользователей для тестирования;
- срок жизни стенда;
- решение, нужны ли резервные копии;
- решение, нужна ли двухфакторная авторизация;
- решение, нужно ли ограничивать доступ по IP.

## 5. Требования к VPS

Рекомендуемый вариант:

- провайдер: предварительно JustHost или другой VPS-провайдер с подходящими характеристиками;
- ОС: Ubuntu Server LTS;
- процессор: 4 vCPU;
- оперативная память: 8 GB;
- диск: 60 GB SSD;
- swap: включен;
- публичный IPv4;
- доступные порты 80 и 443;
- SSH-доступ для настройки.

50 GB диска можно рассматривать только как нижнюю границу для короткой демонстрации. Рекомендуемый вариант - 60 GB SSD.

## 6. Требования к домену и DNS

Wildcard DNS не нужен. Достаточно шести обычных A-записей.

Основные адреса Hermes-сред:

- `node1.example.com -> IP сервера`
- `node2.example.com -> IP сервера`
- `node3.example.com -> IP сервера`

Адреса страниц авторизации:

- `auth-node1.example.com -> IP сервера`
- `auth-node2.example.com -> IP сервера`
- `auth-node3.example.com -> IP сервера`

Первые три адреса ведут в три Hermes-среды. Вторые три адреса используются для страниц входа в каждую среду.

## 7. Авторизация

Для каждой Hermes-среды будет отдельная система входа. Это снижает риск пересечения пользователей и сессий между средами.

Технически для авторизации используется Authelia. Authelia - это сервис, который показывает страницу входа и проверяет пользователя.

Единая корпоративная авторизация будет рассматриваться позже, в следующей версии.

## 8. Изоляция сред

Каждая Hermes-среда будет иметь:

- свои данные;
- свои настройки;
- свои журналы работы;
- свои секреты и ключи;
- свои пользовательские сессии;
- свой внутренний контейнерный контур.

Это означает:

- пользователь одной среды не получает автоматический доступ к другой;
- данные разных сред не смешиваются;
- журналы работы разных сред не смешиваются;
- перезапуск одной среды не должен ломать остальные.

Docker используется как контейнерная основа для запуска сервисов. Traefik используется как входной шлюз: он принимает HTTPS-запросы и направляет пользователя в нужную Hermes-среду.

## 9. Безопасность

Для стенда закладываются базовые правила безопасности:

- доступ только по HTTPS;
- вход только после авторизации;
- наружу открываются только стандартные веб-порты 80 и 443;
- секреты, пароли и ключи не хранятся в репозитории;
- прямой доступ к внутренним сервисам не публикуется;
- каждая среда имеет отдельный контур авторизации.

## 10. Что входит в текущий этап

В текущий этап входит:

- подготовка стенда на новом VPS;
- настройка трех Hermes-сред;
- настройка доменов и HTTPS;
- настройка отдельной авторизации на каждую среду;
- базовая документация по доступу и эксплуатации;
- проверка, что среды изолированы друг от друга.

## 11. Что не входит в текущий этап

В текущий этап не входит:

- единая корпоративная админ-панель;
- создание новых сред из интерфейса;
- общая корпоративная авторизация;
- роли и права уровня enterprise;
- учет токенов;
- биллинг;
- квоты;
- централизованный аудит;
- интеграция с LDAP/Active Directory.

Эти функции относятся к следующему этапу - полноценной корпоративной версии Hermes.

## 12. Что будет считаться успешным результатом

Стенд можно считать успешно подготовленным, если:

- открываются `node1`, `node2` и `node3`;
- все адреса работают по HTTPS;
- без логина попасть в среды нельзя;
- у каждой среды отдельный вход;
- данные и журналы работы не смешиваются;
- перезапуск одной среды не ломает остальные;
- после перезапуска сервера стенд поднимается обратно;
- заказчику переданы адреса, доступы и краткая инструкция.

- Кто предоставляет SSH-доступ?
- Сколько пользователей будет тестировать стенд?
- Нужна ли двухфакторная авторизация?

- Какой срок жизни стенда?
- Нужны ли резервные копии?
- Нужно ли сохранять данные после завершения тестирования?
- Кто будет администрировать стенд после запуска?

## 14. Следующий этап

После тестирования стенда можно будет сформировать требования к полноценной корпоративной версии Hermes.

В следующий этап могут войти:

- единая админ-панель;
- управление пользователями;
- управление нодами;
- общая авторизация;
- роли и права;
- лимиты;
- аудит;
- мониторинг;
- автоматическое создание новых сред.
