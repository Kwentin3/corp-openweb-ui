# Security Minimum

## Минимальные правила PRD-0

- HTTPS only.
- OpenWebUI не публикуется напрямую наружу.
- Публичный perimeter ограничен `22/tcp`, `80/tcp`, `443/tcp`.
- Docker не публикует наружу ничего, кроме Traefik `80/443`; OpenWebUI не имеет прямого public port.
- UFW включен с default deny incoming и default allow outgoing.
- Fail2ban установлен, active, `sshd` jail enabled.
- Warning banner включен и виден пользователям.
- Публичная регистрация отключена или пользователи попадают в `pending`.
- Один администратор, 3-4 обычных пользователя.
- OpenAI и Gemini API-ключи доступны только администратору/серверу.
- `WEBUI_SECRET_KEY` задается явно, хранится только в `.env`/password manager и не меняется между пересозданиями контейнера.
- `CORS_ALLOW_ORIGIN` ограничен `https://gpt.alpha-soft.ru`.
- Реальный `.env` не коммитится.
- Backup с `.env` хранится только server-local.
- В публичном репозитории не хранить точный SSH endpoint, public IP, операторские IP и персональные данные пилотных участников.

## Provider security

- Primary provider key хранится в server-local `.env`.
- Secondary provider key вводится через OpenWebUI Admin UI.
- Пользователям не передаются API keys.
- Model ids можно фиксировать в docs/deployment note, если они не являются секретами.
- Billing/quota/region проверяет оператор до пилота.

## Пользовательские ограничения

Пользователям запрещено отправлять в чат:

- пароли;
- токены;
- API-ключи;
- приватные SSH-ключи;
- OAuth secrets;
- коммерчески чувствительные документы без отдельного разрешения.

## Warning banner

Warning banner является частью минимальной пользовательской защиты PRD-0. Он должен напоминать, что нельзя отправлять пароли, токены, API-ключи, приватные SSH-ключи и закрытые персональные данные, а ответы модели нужно проверять.

Баннер не заменяет полноценную политику безопасности, обучение пользователей или ответственность пользователя за вводимые данные.

## Админские действия

Администратор должен:

- создать пользователей вручную;
- проверить, что обычный пользователь не видит API-ключи;
- проверить, что пользователь видит warning banner;
- отключить signup после создания пользователей;
- проверить strict TLS без `curl -k`;
- проверить `bash scripts/network-hardening-check.sh --strict` после deploy;
- проверить backup;
- собрать обратную связь после 1-2 недель.

## Docker/UFW caveat

Docker может управлять iptables и публиковать порты в обход ожидаемой UFW-модели. Поэтому UFW не считается единственной границей защиты. В PRD-0 запрещено публиковать наружу что-либо, кроме Traefik `80/443` и SSH `22`. После `docker compose up -d` обязательна проверка реальных listeners через `ss` и `bash scripts/network-hardening-check.sh --strict`.

После стабилизации пилота администратор может сменить первичный пароль через UI и убрать bootstrap password из активного server-local `.env`, но это не blocker PRD-0.

## Не делаем в PRD-0

Не внедряем SSO/OIDC, сложный RBAC, DLP, WAF, корпоративный аудит, SIEM, fork OpenWebUI, white-label, плагины и frontend-правки. Это future scope после подтверждения потребности.
